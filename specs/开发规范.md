# 开发规范

本文档定义 SELF-BLOG 项目的代码风格、命名约定、Git 工作流和 AI 协作协议，确保代码库长期可维护。

---

## 1. 语言与运行时要求

| 维度       | 规格                                                                            |
| -------- | ----------------------------------------------------------------------------- |
| **语言**   | **TypeScript 5.x**（全栈统一；禁止新增 JavaScript 源码文件，`*.config.*` 除外）                 |
| **运行时**  | **Node.js >= 20**（禁止使用 Node 18/19；CI 与本地版本必须一致）                               |
| **包管理器** | **npm >= 10**（禁止使用 pnpm / yarn）                                               |
| **严格模式** | **必须开启 `strict: true`**；**禁止 `any`**；允许的唯一例外：第三方类型缺失时使用 `unknown` 并在同文件写明替代计划 |

## 4. TypeScript 特定规范

### 4.1 类型导入（必须使用 `import type`）

```ts
// ✅ 正确
import type { BlogPost } from '@/shared/types/content';

// ❌ 错误：会引入运行时依赖/影响 tree-shaking
import { BlogPost } from '@/shared/types/content';
```

### 4.2 `interface` vs `type`

| 场景             | 偏好          | 原因               |
| -------------- | ----------- | ---------------- |
| **对象形状（可被扩展）** | `interface` | 可合并声明，适合“数据结构契约” |
| **联合/交叉/工具类型** | `type`      | 表达力更强            |

### 4.3 避免的写法

```ts
// ❌ 避免：any
export function parse(input: any) {
  return input.title;
}

// ✅ 替代：unknown + narrowing
export function parse(input: unknown) {
  if (!input || typeof input !== 'object') return null;
  if (!('title' in input)) return null;
  return (input as { title: string }).title;
}
```

```ts
// ❌ 避免：非空断言
const slug = params!.slug;

// ✅ 替代：显式校验 + 早返回
if (!params?.slug) return notFound();
const slug = params.slug;
```

```ts
// ❌ 避免：隐式 any（关闭 strict 的代价）
export function formatDate(date) {
  return String(date);
}

// ✅ 替代：显式类型
export function formatDate(date: Date): string {
  return date.toISOString();
}
```

## 5. Next.js / React 特定规范

### 5.1 组件设计原则

* **必须默认 Server Component**：只有在需要浏览器交互（事件、状态、DOM API）时才写 `'use client'`。
* **禁止在 Client Component 中读取文件系统内容**：内容读取只能在 Server 层（构建期/服务端）完成，再下发到客户端。
* **页面只做编排**：`src/app/**/page.tsx` 只负责“组装模块输出 + 渲染 UI”，禁止在页面内写复杂数据整形。

### 5.2 组件文件结构

```tsx
// 1) 'use client'（仅当必须时出现；且必须在第一行）
// 'use client';

// 2) type 导入
import type { BlogPost } from '@/shared/types/content';

// 3) 依赖导入（第三方 → 项目内；组间必须空行）
import Link from 'next/link';

import { cn } from '@/shared/lib/cn';

// 4) 类型/常量
type Props = {
  post: BlogPost;
};

// 5) 组件定义（必须命名导出）
export function ArticleCard({ post }: Props) {
  return (
    <Link className={cn('block')} href={`/blog/${post.slug}`}>
      {post.title}
    </Link>
  );
}
```

### 5.3 导入导出规则

* **必须使用命名导出**：`export function X()` / `export const X = ...`。
* **禁止默认导出**：`export default ...`。
* **例外**：Next.js `page.tsx`/`layout.tsx` 按框架约定可使用默认导出（若选择默认导出，则同目录禁止再出现同名命名导出页面组件）。

### 5.4 导入排序

```ts
// 顺序必须为：1) type  2) 第三方  3) 项目内（绝对路径）  4) 相对路径
// 组间必须空一行
import type { X } from '@/shared/types/x';

import Link from 'next/link';

import { y } from '@/shared/lib/y';

import { z } from './z';
```

### 5.5 Hooks 使用规范

* **自定义 Hooks 命名必须以 `use` 开头**：`useSearch()`。
* **Hooks 存放位置**：

  * 业务 hooks：`src/modules/<module>/hooks/`
  * 通用 hooks：`src/shared/hooks/`
* **禁止在 hooks 中读写 `content/` 文件**：任何文件系统 IO 必须在服务端模块完成。

## 6. 样式规范

### 6.1 唯一样式方案

* **唯一方案**：Tailwind CSS。
* **动态类名**：必须通过 `cn()`（或等效合并函数）拼接，禁止手写 `condition ? 'a' : 'b'` 的长串表达式。
* **响应式策略**：使用 Tailwind 断点前缀（`sm/md/lg/xl`），禁止写自定义媒体查询（除非必须且写在 `src/shared/styles/`）。

```tsx
// ✅ 正确：cn 合并
<div className={cn('text-sm', isActive && 'font-semibold')} />
```

### 6.2 禁止的样式写法

```tsx
// ❌ 禁止：内联 style（除非是运行时数值且无法用 Tailwind 表达）
<div style={{ marginTop: 12 }} />

// ❌ 禁止：大段三元拼接导致不可读
<div className={isA ? '...' : isB ? '...' : '...'} />

// ✅ 正确：拆成语义片段 + cn
<div className={cn('mt-3', variant === 'primary' && 'text-black')} />
```

## 7. Git 工作流

### 7.1 分支策略（独立开发者：简化 Trunk-Based）

```
main ──●────────●────────●──→（始终可部署）
        ╲
         ╲─ feature/xxx ─●─┘（短分支，合并回 main）
```

| 规则         | 说明                                |
| ---------- | --------------------------------- |
| **主分支**    | `main`，必须始终可部署                    |
| **功能分支**   | 从 `main` 拉出，合并回 `main`            |
| **分支生命周期** | 必须 ≤ 3 天；超过 3 天必须拆分               |
| **合并方式**   | 必须 Squash Merge（保证 `main` 提交历史干净） |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型          | 用途     | 示例                        |
| ----------- | ------ | ------------------------- |
| `feat/`     | 新功能    | `feat/blog-mdx-render`    |
| `fix/`      | Bug 修复 | `fix/search-index-empty`  |
| `refactor/` | 重构     | `refactor/content-loader` |
| `docs/`     | 文档更新   | `docs/dev-standards`      |
| `chore/`    | 构建/配置  | `chore/eslint-prettier`   |

### 7.3 提交信息规范 (Conventional Commits)

提交信息**必须**遵循：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型         | 说明            |
| ---------- | ------------- |
| `feat`     | 新功能           |
| `fix`      | Bug 修复        |
| `refactor` | 重构（不改变功能）     |
| `style`    | 代码格式调整（无逻辑变更） |
| `docs`     | 文档变更          |
| `test`     | 测试相关          |
| `chore`    | 构建/工具/依赖      |
| `perf`     | 性能优化          |
| `ci`       | CI/CD 配置      |

#### 作用域 (scope)

| scope        | 对应范围           |
| ------------ | -------------- |
| `about`      | 关于我模块          |
| `projects`   | 作品集模块          |
| `blog`       | 博客模块           |
| `search`     | 站内搜索模块         |
| `publishing` | 本地编辑/发布链路      |
| `content`    | 内容解析/内容模型层     |
| `shared`     | 共享组件/工具        |
| `specs`      | 规格文档（`specs/`） |

#### 提交信息规则

* `subject` **必须**使用英文小写动词开头，且 **必须** ≤ 72 字符。
* `subject` **禁止**句号结尾。
* `body` **必须**解释“为什么”，禁止逐条复述改了哪些文件。
* `footer` 如关联任务/Issue，必须用 `Refs: #123` 或 `Closes: #123`。

#### 示例

```
feat(blog): add mdx frontmatter validation

The blog content is repo-managed and must be safe to build deterministically. This adds a build-time validator to prevent missing required fields from breaking production builds.

Refs: #12
```

```
fix(search): avoid empty index when no posts exist

When the content directory is empty, the index generator produced an invalid payload, causing the search page to crash at runtime. This makes the generator emit a valid empty index and keeps the UI stable.
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释                           | 不需要注释           |
| ------------------------------ | --------------- |
| **业务规则的“原因”**（例如：为什么要对草稿隐藏）    | 简单赋值/渲染逻辑       |
| **与外部系统/平台约束**（例如：Vercel 构建限制） | 直观的工具函数         |
| **非直觉的边界条件**（例如：slug 冲突处理策略）   | 显而易见的 if/for    |
| **安全相关约束**（例如：为什么不能在客户端暴露字段）   | 组件 props 的自解释命名 |

### 8.2 注释风格

```ts
// ✅ 解释“为什么”：选择构建期索引以避免引入服务端状态与数据库成本
export const SEARCH_INDEX_SOURCE = 'build-time';

// ❌ 描述“做了什么”：这个常量是搜索索引来源
export const SEARCH_INDEX_SOURCE_BAD = 'build-time';
```

### 8.3 TODO / FIXME / HACK 标记

```ts
// TODO(#issue-id): <一句话目标> — <限制/依赖>
// FIXME(#issue-id): <当前 bug 的外显> — <根因>
// HACK(#issue-id): <为什么是临时方案> — <替换方案>
```

## 9. 错误处理规范

### 9.1 服务端错误处理

* **Server 层必须返回可控错误形状**：禁止把原始异常对象直接返回到 UI。
* **内容读取/解析失败必须 fail-fast**：构建期校验失败必须直接失败（阻止发布），禁止静默忽略。

```ts
export type AppError =
  | { kind: 'validation'; message: string }
  | { kind: 'not_found'; message: string }
  | { kind: 'unexpected'; message: string; requestId: string };
```

### 9.2 客户端错误处理

* UI 提示 **必须**用户可理解，**禁止**展示 stack trace。
* 页面级错误 **必须**通过 Next.js `error.tsx`（实现阶段落盘）统一兜底。

### 9.3 安全原则

* UI **禁止**展示原始异常堆栈、路径、环境变量、构建机信息。
* 表单校验 **必须**双重执行：客户端（体验）+ 服务端（安全）。

## 11. 依赖管理

### 11.1 安装规范

```bash
# ✅ 正确
npm install <pkg>
npm install -D <pkg>

# ❌ 禁止
pnpm add <pkg>
yarn add <pkg>
npm install <pkg>@<手写版本号>
```

### 11.2 依赖审查

* 新依赖 **必须**满足：维护活跃、体积可控、许可允许、替代方案不足以解决问题。
* **禁止**引入“与当前架构冲突”的依赖：例如需要服务器数据库的搜索服务、在线 CMS（与“无账号、静态化为主”冲突）。

### 11.3 版本锁定

* `package-lock.json` **必须提交**。
* 升级依赖 **必须**在独立分支完成，且提交信息必须为 `chore(deps): ...`。

## 14. 目录依赖方向（速查）

```
src/app/         → src/modules/* → src/content/ → src/shared/
                 ↘ (允许)         ↗

允许：
  src/app/       → src/modules/*
  src/modules/*  → src/shared/*
  src/modules/*  → src/content/*
  src/content/   → src/shared/*

禁止：
  src/shared/*   ✗→ src/modules/*
  src/shared/*   ✗→ src/app/*
  src/content/   ✗→ src/modules/*
  src/modules/A  ✗→ src/modules/B（除非通过 src/shared/ 明确抽象出共用能力）
```

> 详细放置规则见 `specs/项目结构.md`。

## 15. 安全编码规范

| 类别       | 规则                                                                                     |
| -------- | -------------------------------------------------------------------------------------- |
| **输入校验** | 所有来自 URL 参数/表单输入/内容 frontmatter 的字段必须校验；校验失败必须返回可控错误形状                                 |
| **注入防护** | 禁止拼接 HTML 字符串渲染；渲染 MDX 时必须使用可信编译链路；禁止 `dangerouslySetInnerHTML`（除非经过严格 sanitize 且记录原因） |
| **XSS**  | 任何用户可控内容（包括 Markdown）在进入 DOM 前必须经过安全渲染链路；禁止在客户端拼接不可信 HTML                              |
| **密钥管理** | 任何密钥必须仅存在于服务端环境变量；禁止写入仓库；禁止写进 `NEXT_PUBLIC_`                                           |
| **文件写入** | 本地编辑器写文件能力必须只在 `development` 启用；生产环境必须不可达                                              |
| **认证检查** | 本项目 MVP 不做账号体系；任何“需要登录”的实现均为越界（必须拒绝）                                                   |
