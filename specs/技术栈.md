# 技术栈选型

## 1. 核心决策 (Key Decisions)
结合 `specs/产品概述.md` 的约束（Vercel 部署、静态化为主、无账号体系、本地可视化编辑+一键发布、极简文字优先、MVP 需要站内搜索与基础 SEO），推荐采用“**内容文件在仓库（Markdown/MDX）+ 构建期生成（SEO/索引）+ Vercel 持续部署**”的路线：

- **选择 Next.js（App Router）**：最贴合 Vercel 平台能力，SSR/SSG/ISR 可按页面需要取舍；对 SEO、站点地图、OG 等支持成熟；未来扩展（RSS/统计/评论）也不需要推倒重来。
- **内容源放在 Git 仓库**：用 Markdown/MDX 与结构化元数据（frontmatter / JSON / YAML）承载“关于我/作品/博客”，天然适配“本地编辑 + 一键发布”，且长期维护成本最低。
- **站内搜索在构建期生成索引**：不引入后端与数据库；将文章/作品索引打包为静态 JSON，前端用轻量搜索库做检索即可。

备选方案（不作为默认）：
- **Astro**：纯静态站体验更轻，适合极简博客；但“本地可视化编辑器 + 与内容模型的深度联动”一般需要额外工程组织。若你确定不需要任何 SSR/动态能力，可作为第二选择。

## 2. 前端 (Frontend)
- **框架**: Next.js（App Router）+ React + TypeScript
- **样式方案**: Tailwind CSS（配合少量 `shadcn/ui` 组件用于表单/编辑器 UI；站点展示仍以极简文本排版为主）
- **状态管理**: 优先 React 内建能力（Server Components + URL state + `useState`），必要时用 Zustand（仅在编辑器/复杂交互出现时引入）
- **内容渲染**: MDX（文章/作品详情），支持代码高亮与自定义组件（Callout、图注等）
- **SEO**: Next.js Metadata API + `next-sitemap`（生成 sitemap/robots），OG 图片可后续扩展（可选 `@vercel/og`）
- **站内搜索**: 构建期生成索引（JSON）+ 前端搜索库（推荐 MiniSearch；备选 FlexSearch）

## 3. 后端 (Backend)
MVP 阶段建议 **不做传统后端**（符合“无账号体系、静态化为主、低维护成本”）。

- **运行时/语言**: Node.js（随 Next.js）
- **Web 框架**: Next.js Route Handlers（仅在需要时使用，比如本地编辑器的文件写入/校验接口、或构建期工具 API）
- **API 协议**: REST（简单明确；无必要引入 GraphQL）

> 说明：如果未来引入“评论/统计/订阅/RSS 自动发布”等能力，再补充独立后端或托管服务即可，不影响当前架构。

## 4. 数据存储 (Database)
MVP 阶段：
- **主数据源**: Git 仓库中的内容文件（`content/` 目录内的 Markdown/MDX + frontmatter），以及少量结构化数据文件（JSON/YAML）承载“关于我/简历/项目列表”等。
- **数据库**: 暂不需要
- **缓存**: 暂不需要

未来需要“动态数据/多来源聚合/可编辑元数据”等场景时：
- **主数据库（备选）**: PostgreSQL
- **ORM（备选）**: Prisma

## 5. 基础设施 & 工具 (Infra & Tools)
- **部署**: Vercel（强约束）
- **代码托管/发布链路**: GitHub + Vercel Git 集成（`main` 分支自动构建部署；回滚用 Vercel Deployments）
- **包管理**: npm
- **质量与规范**: ESLint + Prettier + TypeScript（必要时加 `lint-staged`/`husky` 做本地提交前检查）
- **内容处理**: Contentlayer（或同类 content collections 工具）在构建期把 MDX/元数据转成类型安全的内容模型，并产出搜索索引输入

## 6. “本地可视化编辑 + 一键发布”建议落地方式（与技术栈强相关）
为满足“不做在线后台/不做账号体系”，推荐把编辑能力做成 **仅本地可用** 的开发工具：

- **本地可视化编辑器**:
  - 方案 A（推荐）：在 Next.js 中提供 `/(local)/editor` 路由，仅在 `development` 环境启用；使用 MDX 富文本编辑器组件（如 MDXEditor）编辑并保存到 `content/`。
  - 方案 B（备选）：独立的本地小工具（Node CLI + 本地 Web UI），专注写入内容文件；站点仍由 Next.js 负责渲染与部署。
- **一键发布**:
  - 通过脚本把“格式化/校验/提交/推送”串起来（例如 `npm publish`），推送后由 Vercel 自动构建发布。
