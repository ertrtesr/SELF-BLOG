# 产品概述

## 1. 项目愿景 (Vision)
*用一句话概括：我们要为谁，提供什么核心价值？*
> 为“主要给自己看、但希望对外体面呈现”的个人创作者，提供一个极简、可持续更新的个人站点，用于集中展示关于我、作品集与博客内容，并能便捷发布到 Vercel。

## 2. 核心价值 (Value Proposition)
*   **用户价值**：用统一风格与结构承载个人信息与内容沉淀；低维护成本地持续更新；可随时对外分享一个稳定、干净的个人主页。
*   **商业/技术价值**：以静态化为主的轻量架构，降低长期运维与成本；通过基础 SEO 与站内搜索提升内容可达性；便于后续渐进扩展（如 RSS/评论/统计）。

## 3. 核心板块 (Core Modules)
*将系统拆解为关键的顶层模块（建议 2-4 个核心板块）。*

### 3.1 关于我 (About)
*   **描述**：承载个人简介与简历信息的展示页，作为对外入口与“我是谁”的统一说明。
*   **关键特性**：
    *   关于我/简历结构化展示（教育、经历、技能等）
    *   与作品集、博客模块的清晰导航联动

### 3.2 作品集 (Projects)
*   **描述**：以项目卡片方式展示代表性作品，强调“我做过什么”和可访问链接。
*   **关键特性**：
    *   项目列表（标题、简介、技术栈、链接、截图/封面）
    *   项目详情页（可选，若 MVP 需要可扩展）

### 3.3 博客 (Blog)
*   **描述**：承载文章列表与文章详情，支持长期写作沉淀，但定位仍以“作品/个人展示”优先。
*   **关键特性**：
    *   文章列表 + 文章详情
    *   基础 SEO（meta、OG、sitemap 等）
    *   站内搜索（对文章/作品进行检索）

### 3.4 内容管理/发布 (Local Publishing)
*   **描述**：以“本地可视化编辑器 + 一键发布”的方式完成内容生产与上线，避免引入网页登录后台与账号体系。
*   **关键特性**：
    *   本地可视化编辑文章与作品内容
    *   一键触发构建与部署到 Vercel（具体机制在后续技术设计阶段确定）

## 4. 目标用户 & 核心场景 (User & Scenes)
### 4.1 目标用户
*   **个人作者（自用优先）**：有内容沉淀与对外展示需求，但不打算做重运营；希望网站“体面、极简、稳定”，维护成本低。
*   **对外查看者（偶发访问）**：通过链接快速了解作者（关于我）并浏览代表性项目与文章；希望加载快、信息结构清晰。

### 4.2 核心场景 (Key Use Cases)
*   **场景 1**：个人作者在本地完成一篇文章/更新一个项目卡片，通过一键发布，使更新上线到 Vercel，并可生成可分享链接。
*   **场景 2**：偶发访问者打开主页，快速理解作者背景（关于我）并浏览作品集；若想深入了解，可通过站内搜索定位到特定文章或项目。

## 5. 关键业务流程 (Core Workflows)
*描述系统最核心的 1-2 条业务链路，这决定了技术架构的复杂度。*
> 内容编辑（本地可视化） -> 生成/更新内容源（如 Markdown/数据文件） -> 构建站点（包含搜索索引与 SEO 产物） -> 部署到 Vercel -> 对外访问与分享

## 6. 验收标准 (Acceptance Criteria)
*项目第一阶段结束的具体标志是什么？（用于防止需求蔓延）*
*   **功能完备性**：
    *   “关于我 / 作品集 / 博客”三大板块可用，信息结构清晰、可导航
    *   支持本地可视化编辑并完成一次“一键发布”上线流程
    *   站内搜索可检索到文章（至少覆盖标题与摘要；覆盖范围可在实现阶段明确）
    *   基础 SEO 生效（至少包含页面 meta/OG 与 sitemap）
*   **技术指标**：
    *   核心页面在常见网络环境下首屏加载体验良好（具体阈值在技术阶段量化，如 LCP/TTFB）
    *   部署在 Vercel，主流程稳定可复现（可重复发布、可回滚）

## 7. 约束条件 (Constraints)
*非功能性的硬性限制。*
*   部署平台：必须为 **Vercel**
*   发布方式：必须支持 **本地可视化编辑 + 一键发布**
*   权限：**不做**登录/后台权限控制（无账号体系）
*   视觉风格：**极简文字优先**
*   MVP 必选能力：**站内搜索**、**基础 SEO**

## 8. 项目边界 (Out of Scope)
*明确本阶段**不做**什么，避免需求蔓延。*
*   不做：网页端后台（类似 WordPress 的在线管理后台）
*   不做：账号登录与权限体系（单用户/多用户均不做）
*   不做：评论系统（如 Giscus/Disqus）
*   不做：访问统计集成（GA/Plausible 等）
*   不做：RSS 订阅
*   不做：从 Notion/语雀/Obsidian 自动同步导入
*   不做：面向运营的增长功能（订阅、邮件推送、复杂 SEO 增长策略等）
